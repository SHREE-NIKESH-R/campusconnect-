1. PROJECT OVERVIEW
================================================================================

CampusConnect is a full-stack web application for managing campus resource
bookings. It allows students and faculty to browse available resources
(classrooms, labs, seminar halls, sports facilities, etc.), make booking
requests, and allows admins to approve/reject bookings and manage resources.

Key Features:
- Role-based access control (Student, Faculty, Admin)
- JWT-based stateless authentication
- Resource browsing with filtering and search
- Booking creation with conflict detection
- Admin dashboard with statistics
- Admin panel for booking approval/rejection
- Admin panel for resource CRUD operations
- Responsive glassmorphism UI design

6. API ENDPOINTS
================================================================================

AUTHENTICATION (/api/v1/auth)
-----------------------------
POST /register          → Register new user (public)
POST /login             → Login, returns JWT token (public)

RESOURCES (/api/v1/resources)
-----------------------------
GET  /                  → List all active resources (public)
GET  /?type={type}      → Filter by resource type (public)
GET  /available         → Find available for time slot (public)
GET  /{id}              → Get resource by ID (public)
POST /                  → Create resource (ADMIN only)
PUT  /{id}              → Update resource (ADMIN only)
PATCH /{id}/toggle-availability → Toggle availability (ADMIN only)
DELETE /{id}            → Soft delete resource (ADMIN only)

BOOKINGS (/api/v1/bookings)
---------------------------
POST /                  → Create booking (authenticated)
GET  /my                → Get my bookings (authenticated)
GET  /all               → Get all bookings (ADMIN only)
GET  /pending           → Get pending bookings (ADMIN only)
PATCH /{id}/status      → Update booking status (ADMIN only)
PATCH /{id}/cancel      → Cancel my booking (authenticated)

DASHBOARD (/api/v1/dashboard)
-----------------------------
GET /stats              → Get admin statistics (ADMIN only)

7. AUTHENTICATION & SECURITY
================================================================================

JWT TOKEN FLOW
--------------
1. User logs in → server validates credentials
2. Server generates access token (24h expiry) + refresh token (7d expiry)
3. Frontend stores token in localStorage as 'cc_token'
4. Every API request includes: Authorization: Bearer <token>
5. JwtAuthenticationFilter validates token on every request
6. If token expired/invalid → 401 → frontend redirects to login

SECURITY CONFIG (SecurityConfig.java)
-------------------------------------
- CSRF disabled (stateless JWT)
- CORS configured for localhost:5173 and localhost:3000
- Public endpoints: /auth/**, /h2-console/**, swagger, GET /resources/**
- Admin-only: POST/PUT/DELETE/PATCH /resources/**, /dashboard/**, /bookings/all, /bookings/pending
- Method security: @PreAuthorize("hasRole('ADMIN')") on controller methods
- Passwords: BCrypt hashing

ROLES
-----
STUDENT  → Can browse resources, create bookings, view own bookings, cancel own bookings
FACULTY  → Same as student + bookings prioritized for approval
ADMIN    → Full access: approve/reject bookings, manage resources, view dashboard stats

8. FEATURE-BY-FEATURE BREAKDOWN
================================================================================

8.1 AUTHENTICATION
------------------
Backend:
  - AuthController handles /login and /register
  - AuthService: validates email uniqueness, hashes password with BCrypt,
    generates JWT tokens via JwtUtils
  - CustomUserDetailsService loads user by email for Spring Security

Frontend:
  - LoginPage: form with email/password + demo account shortcuts (admin/student/faculty)
  - RegisterPage: form with fullName, email, password, department, studentId, phone, role
  - useAuth hook: manages auth state via React Context, stores token/user in localStorage
  - API client attaches Bearer token to every request via interceptor

8.2 RESOURCE BROWSING
---------------------
Backend:
  - ResourceController: GET endpoints for listing, filtering by type,
    finding available for date/time slot, CRUD (admin)
  - ResourceService: business logic for CRUD, availability toggle, soft delete
  - ResourceRepository: JPA queries with active=true filtering

Frontend:
  - ResourcesPage: grid view with search, type filter, availability filter
  - Resource cards show: name, type badge, location, capacity, amenities, availability
  - Click card → modal with full details + "Book Now" button
  - Admin sees all resources; students see only available ones for booking

8.3 BOOKING CREATION
--------------------
Backend:
  - BookingController: POST /bookings with CreateBookingRequest
  - BookingService validates:
    * Resource exists and is available
    * Start time < end time
    * Booking date is not in the past
    * No conflicting bookings for same resource/date/time (findConflictingBookings)
  - Booking saved with status=PENDING

Frontend:
  - BookingPage: 3-column layout on desktop
    * Left: resource selector (cards with availability)
    * Middle: date picker (7-day week view, prev/next week navigation)
    * Right: time slot picker (08:00-19:00), purpose textarea,
      attendees count, notes, submit button
  - Success screen with "View My Bookings" and "Book Another" buttons

8.4 MY BOOKINGS
---------------
Backend:
  - GET /bookings/my returns user's bookings ordered by createdAt DESC
  - PATCH /bookings/{id}/cancel allows users to cancel their own bookings

Frontend:
  - MyBookingsPage: list of booking cards with filter tabs (ALL/PENDING/APPROVED/REJECTED/CANCELLED)
  - Each card shows: resource name, date, time, location, purpose, status badge,
    admin notes (if any), cancel button (for PENDING/APPROVED)

8.5 ADMIN DASHBOARD
-------------------
Backend:
  - DashboardController: GET /dashboard/stats (admin only)
  - DashboardService aggregates: total resources, available resources,
    total bookings, pending bookings, approved bookings, total users,
    today's bookings, recent 10 bookings

Frontend:
  - DashboardPage: stat cards for admin, recent bookings table
  - Non-admin sees role card + quick action stat cards

8.6 ADMIN PANEL - BOOKING MANAGEMENT
------------------------------------
Backend:
  - GET /bookings/all → all bookings sorted by bookingDate ASC, startTime ASC
  - GET /bookings/pending → pending bookings only
  - PATCH /bookings/{id}/status → approve/reject with admin notes

Frontend:
  - AdminPage "Bookings" tab: table with columns User, Resource, Date & Time,
    Purpose, Status, Actions
  - Status filter buttons: ALL, PENDING, APPROVED, REJECTED, CANCELLED
  - PENDING bookings show approve (green check) and reject (red X) buttons
  - Bookings sorted by closest upcoming event first

8.7 ADMIN PANEL - RESOURCE MANAGEMENT
-------------------------------------
Backend:
  - Full CRUD via ResourceController (admin-only endpoints)
  - Soft delete (sets active=false instead of removing from DB)
  - Toggle availability without deleting

Frontend:
  - AdminPage "Resources" tab:
    * Add Resource form: name, type, capacity, location, building, room number,
      description, amenities (comma-separated)
    * Resources table: name, type, location, capacity, status, actions
    * Each row has "Disable/Enable" toggle button

9. DATA FLOW EXAMPLES
================================================================================

EXAMPLE 1: USER LOGIN
---------------------
1. User clicks "Sign in" on LoginPage
2. Frontend: authApi.login({email, password}) → POST /api/v1/auth/login
3. Backend: AuthController.login() → AuthService.login()
4. AuthService: authenticate via AuthenticationManager, generate JWT tokens
5. Response: {accessToken, refreshToken, tokenType, user}
6. Frontend: store token in localStorage, set user in AuthContext, redirect to /dashboard

EXAMPLE 2: CREATE BOOKING
-------------------------
1. User selects resource, date, time, fills purpose on BookingPage
2. Frontend: bookingApi.create(payload) → POST /api/v1/bookings
3. Backend: JwtAuthenticationFilter validates token
4. BookingController.create() → BookingService.createBooking()
5. BookingService validates constraints, checks for conflicts
6. If valid: save Booking with status=PENDING
7. Response: BookingResponse with all details
8. Frontend: show success screen

EXAMPLE 3: ADMIN APPROVES BOOKING
---------------------------------
1. Admin opens AdminPage → Bookings tab
2. Frontend: bookingApi.getAllBookings() → GET /api/v1/bookings/all
3. Admin clicks green checkmark on a PENDING booking
4. Frontend: bookingApi.updateStatus(id, {status: APPROVED, adminNotes})
5. Backend: validates admin role, updates booking status and approvedAt timestamp
6. Frontend refreshes bookings list

EXAMPLE 4: ADD RESOURCE (ADMIN)
-------------------------------
1. Admin opens AdminPage → Resources tab
2. Fills Add Resource form, clicks "Add Resource"
3. Frontend: resourceApi.create(payload) → POST /api/v1/resources
4. Backend: validates admin role, creates Resource with active=true, available=true
5. Frontend: clear form, show success message, reload resources list

11. KNOWN ISSUES & FIXES APPLIED
================================================================================

ISSUE 1: Backend Connection Refused (ECONNREFUSED)
--------------------------------------------------
Symptom: Vite proxy error for /api/v1/resources
Cause: Spring Boot backend was not running
Fix: Start backend via IntelliJ or `mvnw spring-boot:run`

ISSUE 2: Added Resources Not Appearing
--------------------------------------
Symptom: After adding resource via admin panel, it doesn't show in list
Root Cause: Lombok @Builder ignores field initializer values.
            When using @Builder, `active = true` and `available = true`
            in the entity class were NOT being set.
            New resources saved with active=false, available=false.
Fix: Added @Builder.Default annotation to all fields with default values:
     - Resource.java: available, active, amenities, bookings
     - User.java: role, enabled, bookings
     - Booking.java: status, attendeesCount

ISSUE 3: Admin Panel Bookings Not Sorted
----------------------------------------
Symptom: Bookings shown in random order
Fix: Added backend sorting (findAllByOrderByBookingDateAscStartTimeAsc)
     + frontend client-side sort fallback for closest events first

ISSUE 4: Windows PowerShell Command Separator
---------------------------------------------
Symptom: `cd backend && mvnw spring-boot:run` fails with "&& is not valid"
Fix: Use PowerShell syntax: `cd backend; .\mvnw.cmd spring-boot:run`
     Or use Command Prompt instead of PowerShell
